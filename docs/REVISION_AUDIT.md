## 1. Per-file orientation notes

### `README.md`
Project front page. Frames the problem as **HA-LMAPF** (Human-Aware Lifelong
MAPF under Partial Observability) and documents the two-tier architecture,
metrics list, baseline list (GlobalOnly, PIBTOnly, IgnoreHumans, RHCR-like,
WHCA*), experiment groups, and quick-start commands. No code symbols.

### `src/ha_lmapf/simulation/simulator.py`
The central orchestrator. Class **`Simulator`** owns the env, agent/human
dictionaries, planners, allocators, controllers, metrics tracker, and replay
writer; entry points are `step_once()` and `run()`. Also exposes hooks
`plans()`, `decided_next_positions()`, `local_paths()`, `set_local_path()`,
`mark_safety_wait()`, `flag_major_deviation()`, `mark_task_assigned()` etc.
that controllers and resolvers read via the `SimStateView` protocol.

### `src/ha_lmapf/simulation/agent_dynamics.py`
Pure physics function. **`apply_action(env, agent_state, action)`** computes
the next `AgentState` from a `StepAction`, validating only static obstacles
(walls). Dynamic-obstacle and inter-agent collisions are *not* checked here —
they are caught later by the simulator.

### `src/ha_lmapf/local_tier/agent_controller.py`
Tier-2 per-agent reactive controller. Class **`AgentController`** runs a
sense-plan-act loop: build forbidden set from inflated visible humans, follow
global plan if safe, else local A* detour, else WAIT (hard mode) or escape
move (soft mode); finally route through a `BaseConflictResolver`. Helpers
`_global_path_truly_exhausted`, `_global_path_blocked`, `_deviation_ratio`,
`clear_path`. `fallback_wait_limit` triggers a global replan request after N
consecutive waits when local replan is disabled.

### `src/ha_lmapf/local_tier/local_planner.py`
Local A* implementation. Class **`AStarLocalPlanner`** with `plan(env, start,
goal, blocked, guidance_cells=None) -> List[Cell]`; supports hard
(`blocked` impassable) and soft (`BLOCKED_CELL_COST=50`) safety modes,
guidance bias (`GUIDANCE_DEVIATION_COST=1`) toward remaining global-plan
cells, and `MAX_EXPANSIONS=10_000` cap.

### `src/ha_lmapf/local_tier/sensors.py`
Function **`build_observation(agent_id, sim_state, fov_radius) -> Observation`**.
Filters humans/agents inside Manhattan-FOV, builds `blocked = env.blocked ∪
{visible human cells} ∪ {visible agent cells} ∪ {decided_next_positions of
others}`. Note: human cells are added *non-inflated* here; inflation by `B_r`
happens later in the controller.

### `src/ha_lmapf/local_tier/conflict_resolution/base.py`
Defines **`ImminentConflict`** dataclass (`kind`, `other_agent_id`, `cell`,
`edge`, `source`) and the shared detector
**`detect_imminent_conflict(agent_id, desired_cell, sim_state)`**. Order of
checks: (1) decided next-positions of earlier agents this step; (2) other
agents' current cells; (3) other agents' next-step positions per global plan
(skipping `stale_global_plan_agents`). Also exports `BaseConflictResolver`
abstract class with helper `action_toward(cur, nxt) -> StepAction`.

### `src/ha_lmapf/local_tier/conflict_resolution/priority_rules.py`
**`PriorityRulesResolver`** — communication-free deterministic resolver.
Computes priority tuple `(urgency, wait_steps, -agent_id)` (see §6).
Default ctor: `starvation_threshold=10`, `boost=50`, `allow_side_step=True`.
On loss: side-step to a safe neighbor (preferred) or WAIT.

### `src/ha_lmapf/local_tier/conflict_resolution/token_passing.py`
**`TokenPassingResolver`** with internal **`_TokenState`** per contested cell.
Same priority tuple as `PriorityRulesResolver` but **without** the starvation
boost. Includes fairness rotation: after `fairness_k=5` consecutive wins on
the same cell, ownership rotates to the next-best contender. Loser tries
side-step, otherwise WAITs.

### `src/ha_lmapf/humans/safety.py`
Geometry helpers. **`inflate_cells(cells, radius, env)`** returns the
Manhattan-ball of free cells around seed cells (the **B_r** safety buffer).
Also exports **`proximity_penalty(cell, human_cells, radius)`** (soft cost
`max(0, radius - dist + 1)`) and **`forbidden_mask(visible_humans, radius,
env)`** (thin wrapper around `inflate_cells`).

### `src/ha_lmapf/humans/models.py`
Stochastic human motion library. Abstract **`HumanModel.step(env, humans,
rng, agent_positions=None)`** plus 5 concrete subclasses: **`RandomWalkHumanModel`**
(Boltzmann inertia: `beta_go`, `beta_wait`, `beta_turn`),
**`AisleFollowerHumanModel`** (corridor-likelihood φ = -dist-to-obstacle,
`alpha`, `beta`, `wait_penalty`), **`AdversarialHumanModel`**
(target field `gamma * (lambda * bottleneck − (1−lambda) * dist-to-agent)`),
**`MixedPopulationHumanModel`** (categorical per-human type sampling), and
**`ReplayHumanModel`** (deterministic JSON replay, with `from_json` and
`generate_and_record` factories). Helpers `_legal_successors`,
`_softmax_sample`, `_continuation_cell`, distance-field BFS, bottleneck
centrality.

### `src/ha_lmapf/global_tier/rolling_horizon.py`
**`RollingHorizonPlanner`** schedules Tier-1 calls. Triggers: periodic (`step
% replan_every == 0`), `major_deviation` flag, exhaustion
(`stale_global_plan_agents / |agents| ≥ exhaustion_fraction=0.4`), or
safety-wait (`safety_wait_agents / |agents| ≥ safety_wait_fraction=0.3`).
Emergency triggers gated by `_min_emergency_gap = max(3, replan_every//4)`
and a `_last_replan_useful` heuristic. Delegates to a `solver` chosen via
ctor injection or `GlobalPlannerFactory`. Passes `is_lifelong=True` to
solvers that accept it (introspected).

### `src/ha_lmapf/global_tier/planner_interface.py`
**`GlobalPlannerFactory.create(name, **kwargs)`** — string-keyed factory for
all global solvers. Helpers `_wait_path`, `build_wait_plan_for_unassigned`,
and `PlannerWrapper` that pads missing agents with WAIT paths. See §8 for the
solver registry.

### `src/ha_lmapf/core/types.py`
Core dataclasses and enums. Exports **`StepAction`** (UP/DOWN/LEFT/RIGHT/WAIT
enum), **`AgentState`**, **`HumanState`**, **`Task`**, **`TimedPath`** (with
`__call__(step)` clamp behavior), **`PlanBundle`**, **`Observation`**,
**`Metrics`**, **`SimConfig`**. All carry `to_dict()` for replay/JSON logging.

### `src/ha_lmapf/baselines/rhcr_like.py`
**`RHCRPlanner`** — RHCR baseline as a Tier-1-only planner. Two backends:
Python (any `GlobalPlannerFactory` solver, periodic replan every
`replan_every` steps with greedy task allocation) or the official C++
`RHCRSolver` if a `rhcr` / `lifelong` binary is found. Helper
`_check_rhcr_binary_available()`.

### `src/ha_lmapf/baselines/global_only_replan.py`
**`GlobalOnlyController`** — follows the global plan strictly, no local
replanning. WAITs if next planned cell is in `observation.blocked` (which
includes visible humans). Still routes through a `ConflictResolver`. Factory
`make_global_only_controllers(sim_state, conflict_resolver, _fov_radius,
_safety_radius)`.

### `src/ha_lmapf/baselines/pibt_only.py`
**`PIBTOnlyController`** — decentralized greedy: ignores global plans,
chooses neighbor minimizing Manhattan distance to goal (deterministic
tie-break), routes through `PriorityRulesResolver`. Helper
`_check_pibt2_binary_available()` (file-only check; not used by the
controller class itself).

## 5. Where the safety buffer **B_r** is computed and consumed

**Computed (single canonical implementation):**
- `src/ha_lmapf/humans/safety.py:21` — `inflate_cells(cells, radius, env) -> Set[Cell]`
  enumerates the Manhattan diamond of `radius` and keeps only `env.is_free`
  cells.
- `src/ha_lmapf/humans/safety.py:125` — `forbidden_mask(visible_humans, radius,
  env)` is a thin wrapper that extracts `{h.pos for h in visible_humans}` and
  delegates to `inflate_cells`.

**Consumed (call sites):**
1. `src/ha_lmapf/local_tier/agent_controller.py:65` — controller's
   per-step decision, using **only `observation.visible_humans`** (FOV-limited).
   Result is unioned with `observation.blocked` to produce the local
   `forbidden`/`blocked` set passed to the local A* planner. This is the
   *online, partially observable* B_r used for action selection and hard-safety
   enforcement (`agent_controller.py:200`).
2. `src/ha_lmapf/simulation/simulator.py:824-825` — post-step audit, using
   **all simulator-wide humans** (`{h.pos for h in self.humans.values()}`).
   This is the *ground-truth* B_r used to count `safety_violations`.

The local planner itself does not call `inflate_cells`; it receives the
already-inflated `blocked` set as an argument
(`local_planner.py:60-93` — `plan(..., blocked, guidance_cells=None)`).

## 7. Exact priority tuple in `PriorityRulesResolver`

From `priority_rules.py:55-67`:

```python
def _priority(self, agent_id, sim_state) -> Tuple[int, int, int]:
    a = sim_state.agents[agent_id]
    if a.goal is None:
        dist = 10 ** 9
    else:
        dist = manhattan(a.pos, a.goal)
    urgency = -dist
    if a.wait_steps > self.starvation_threshold:   # default threshold=10
        urgency += self.boost                       # default boost=50
    return (urgency, int(a.wait_steps), -int(agent_id))
```

**Tuple:** `(urgency, wait_steps, -agent_id)` — higher tuple wins; ties broken
by lower `agent_id` (because `-agent_id` is larger).

- `urgency = -manhattan(pos, goal)` (or `-1_000_000_000` if no goal),
  optionally `+50` after `wait_steps > 10`.
- Defaults: `starvation_threshold=10`, `boost=50`, `allow_side_step=True`.

`TokenPassingResolver._priority` uses the same tuple **without** the
starvation boost (`token_passing.py:99-107`).

## 8. Decision-table writes vs. next-agent reads

**Mechanism.** `Simulator._decided_next_positions: Dict[int, Cell]` is the
shared decision table.

**Write timing** (`simulator.py:1007-1020`):
1. Cleared once at the top of the per-agent decide loop (line 1007).
2. Within the loop, each agent (in sorted `aid` order) computes its action,
   then **immediately** writes its intended next position to the table
   (`_decided_next_positions[aid] = next_pos`, line 1020).

**Read timing.**
- Inside the same loop, `controllers[aid].decide_action(...)` (line 1013)
  delegates to the conflict resolver, which calls
  `detect_imminent_conflict(...)` (`conflict_resolution/base.py:41`). That
  function reads `sim_state.decided_next_positions()` and treats other agents'
  entries as committed conflicts.
- `sensors.build_observation` also reads the table
  (`sensors.py:46-49`), but observations are built *before* the loop clears
  the table (lines 1001-1003 vs. 1007), so during the decide loop, the
  observations only reflect the **previous step's** decided positions.

**Answer.** *Decision-table writes happen BEFORE the next agent reads.* In the
sorted-aid sequential loop, each agent commits its decision to the table the
moment its action is chosen, and the next agent's `detect_imminent_conflict`
sees that commitment when it executes. Caveat: the per-agent `Observation`
built at step 5a does **not** see this round's decisions — only
`detect_imminent_conflict` (via the resolver) does. This is a correctness
asymmetry to be aware of in the revision.

## 11. Theorem 1 Invariant Audit

Theorem 1 (paper Section 4.5) states that under Algorithm 2 no executed
action is **agent-attributable** to a buffer violation.  The proof depends
on every code path that selects an action respecting the forbidden set
**F = B_{r_safe}(X_t^{Phi_i}) ∪ D(t)_extended** — including the conflict
resolver's loser fallback.  This section traces each code path and records
whether it respects F at the time of audit (commit `936bea4`, before any
fix in this section is applied).

### Where F is computed (controller side)

`src/ha_lmapf/local_tier/agent_controller.py:64-66`:

```python
human_cells = {h.pos for h in observation.visible_humans.values()}
forbidden = inflate_cells(human_cells, radius=int(self.safety_radius), env=sim_state.env)
blocked = set(observation.blocked) | set(forbidden)
```

The controller correctly assembles `blocked = F ∪ D(t)_extended` and uses
it for the local-A* call at `agent_controller.py:112-115`:

```python
detour = self.local_planner.plan(
    sim_state.env, start=cur, goal=goal, blocked=blocked,
    guidance_cells=guidance_cells,
)
```

**Verdict — controller's local-A* path: F is respected.** Hard-safety
enforcement at `agent_controller.py:200-202` also explicitly checks
`desired_next in forbidden` before delegating to the resolver.

### Where the resolver is invoked

`src/ha_lmapf/local_tier/agent_controller.py:209-217`:

```python
action = self.conflict_resolver.action_toward(...)
resolved_action = self.conflict_resolver.resolve(
    agent_id=aid,
    desired_cell=desired_next,
    sim_state=sim_state,
    observation=observation,
    rng=rng,
)
```

**The resolver does NOT receive `forbidden` (F).**  It only receives
`observation`, whose `.blocked` field per `sensors.py:39-49` is

```
static obstacles ∪ visible human cells ∪ visible agent cells
                ∪ decided next positions (of others, from previous step)
```

— i.e. **D(t)_extended without the r_safe inflation.**  Anything the
resolver does with `observation.blocked` is therefore F-blind.

### `PriorityRulesResolver` loser fallback

`priority_rules.py:43-53`:

```python
if p_self > p_other:
    return self.action_toward(sim_state.agents[agent_id].pos, desired_cell)

# Lose: wait or side-step safely
if self.allow_side_step:
    side = self._safe_side_step(agent_id, sim_state, observation)
    if side is not None:
        return self.action_toward(sim_state.agents[agent_id].pos, side)

return StepAction.WAIT
```

`_safe_side_step` (`priority_rules.py:69-84`):

```python
cur = sim_state.agents[agent_id].pos
for nb in neighbors(cur):
    if not sim_state.env.is_free(nb):
        continue
    if nb in observation.blocked:        # <-- F-blind: F not in observation.blocked
        continue
    c = detect_imminent_conflict(agent_id, nb, sim_state)
    if c is None:
        return nb
return None
```

**Verdict — PriorityRulesResolver loser side-step: F is NOT respected.**  A
losing agent can be steered into a cell adjacent to an observed human if
that cell is otherwise free and unclaimed.  This breaks Theorem 1.

### `TokenPassingResolver` non-owner fallback

`token_passing.py:90-97`:

```python
if winner == agent_id:
    return self.action_toward(...)

# Non-owner: yield or side-step safely
side = self._safe_side_step(agent_id, sim_state, observation)
if side is not None:
    return self.action_toward(sim_state.agents[agent_id].pos, side)
return StepAction.WAIT
```

`_safe_side_step` (`token_passing.py:119-138`) is structurally identical
to PriorityRules' version — same `if nb in observation.blocked: continue`
filter.

**Verdict — TokenPassingResolver non-owner side-step: F is NOT respected.**
Same bug as PriorityRules; same Theorem 1 violation.

### `MAX_EXPANSIONS` constant

The user's task description mentions `MAX_EXPANSIONS=500`.  The actual
value at `local_planner.py:47` is **`MAX_EXPANSIONS = 10_000`**.  When
local-A* exceeds the bound it returns `[]`, after which the controller
falls back to WAIT (hard safety) or escape (soft safety).  Neither
fallback in the controller can violate F in hard-safety mode, so this
path is correct.  Only the resolver path is buggy.

### Summary

| Code path | Respects F? |
|---|---|
| `AgentController` hard-safety check on global plan (`agent_controller.py:200`) | **Yes** |
| `AgentController` local-A* detour (`agent_controller.py:112`) | **Yes** |
| `AgentController` soft-safety escape (`agent_controller.py:148`) | **Yes** (filter at `_find_escape_move`) |
| `PriorityRulesResolver` winner branch | n/a (no movement on loss) |
| `PriorityRulesResolver` loser side-step | **No — bug** |
| `PriorityRulesResolver` loser WAIT fallthrough | Yes (WAIT keeps cur, which is not in F by upstream invariant) |
| `TokenPassingResolver` non-owner side-step | **No — bug** |
| `TokenPassingResolver` non-owner WAIT fallthrough | Yes |
| `PIBTResolver` (out of scope) | also F-blind side-step; not used by paper's approach |

The fix applied in this revision: plumb `forbidden` (F) and a
`local_planner` reference through the resolver call so that
(i) `_safe_side_step` filters against `observation.blocked ∪ F` and
(ii) when no immediate side-step exists the resolver runs a local-A*
detour with `blocked = observation.blocked ∪ F ∪ winner-claimed cells`,
falling through to Safe Wait if A* returns no path.  See
`tests/test_theorem1_resolver.py` for the unit-level invariant and
`tests/test_theorem1_stress.py` for a 200-step end-to-end check.

## 12. RHCR exogenous-blindness audit (paper Section 5.5)

The paper claims that RHCR's agent-attributable buffer-violation count
grows super-linearly because RHCR is exogenous-blind: the Tier-1 solver
plans on the static map only, and the executor has no Tier-2 detour.
This audit verifies the claim against the implementation.

### Tier-1 (planner) is exogenous-blind ✓

`baselines/rhcr_like.py::RHCRPlanner.step` calls

```python
plan = self.solver.plan(
    env=sim_state.env,
    agents=sim_state.agents,
    assignments=assignments,
    step=cur_step,
    horizon=self.horizon,
    rng=None,
)
```

— and the solver wrappers (`global_tier/solvers/rhcr_wrapper.py`,
`lacam_official_wrapper.py`, etc.) only see the static `env`.  No
exogenous-agent positions are propagated to the solver as obstacles,
edge weights, or vertex constraints.  The same holds for the
factory-created `RHCRSolver` invoked directly via `global_solver="rhcr"`
in `SimConfig`.

### Tier-2 (controller) was NOT exogenous-blind — patch applied

`Simulator.__init__` historically wired every agent through
`AgentController` regardless of `global_solver`, so RHCR runs were
actually getting:

* hard-safety enforcement against the inflated buffer F (`agent_controller.py:200`),
* local A* repair around visible exogenous agents (`agent_controller.py:112`),
* resolver-side A* fallback that respects F (`priority_rules.py::_astar_fallback`).

Together these would defeat the paper's "RHCR is blind to exogenous
agents" framing — the controller silently absorbs human awareness and
Theorem 1 starts to apply, hiding RHCR's true violation curve.

**Patch.**  `SimConfig` now carries
`controller_kind: Literal["default", "global_only"] = "default"`
(`core/types.py`).  When set to `"global_only"`, `Simulator.__init__`
instantiates `GlobalOnlyController` (paper baseline mode) instead of
`AgentController`.  The `make_rhcr_blind_config` helper in
`baselines/pibt2_fr.py` returns a `SimConfig` with
`global_solver="rhcr"` *and* `controller_kind="global_only"`, giving
the paper's end-to-end exogenous-blind RHCR.

`GlobalOnlyController` (`baselines/global_only_replan.py`):

* reads `desired_next` from `sim_state.plans()`;
* WAITs only at exact occupied cells via `observation.blocked` (which
  per `sensors.py:39-49` does **not** include the r_safe inflation);
* never calls the local A* planner;
* delegates agent-agent conflicts to the configured resolver.

This produces the expected behaviour: agents walk freely into r_safe
buffer cells around visible exogenous agents, generating
agent-attributable violations as the paper predicts.

### Empirical verification

`tests/test_rhcr_blind.py::test_blind_controller_walks_into_buffer_with_working_solver`
runs a 50-step lifelong simulation on a 5×14 single-corridor map with
a 5-cell exogenous-agent cluster planted mid-corridor.  The
`controller_kind="global_only"` configuration produces strictly
positive `violations_agent_attributable`, i.e. the rigid follower
walks into the buffer.

The companion test
`tests/test_rhcr_blind.py::test_rhcr_blind_walks_into_buffer` exercises
the same property using the actual `rhcr` C++ binary.  On this CI
image the binary segfaults during planning (rc=-11) — a build
artefact, not an algorithmic issue — and the test skips with a
diagnostic.  See `docs/SOLVER_STATUS.md` for the broader binary
matrix.

### Other baselines that share the rigid-follower path

The same `controller_kind="global_only"` switch powers the **PIBT2-FR**
baseline (paper Section 5.5: PIBT2 with R=1, no Tier-2):
`make_pibt2_fr_config` in `baselines/pibt2_fr.py`.  The **No-Buffer**
ablation (`make_no_buffer_config`) does NOT use the rigid follower —
it keeps the full architecture and just sets `safety_radius=0`,
verifying that Theorem 1 holds even when the buffer collapses to the
exogenous agents' exact cells (`tests/test_baseline_no_buffer.py`
passes with `violations_agent_attributable == 0`).

## 13. Safety-violation attribution — WAIT-counterfactual rewrite

**Problem.** The pre-rewrite classifier in
`simulator.py::_detect_collisions_and_near_misses` re-tested the
local planner's own constraint instead of measuring an independent
quantity:

```python
observed_positions = [h.pos for h in humans_at_decision.values()
                     if L1(a_prev, h.pos) <= fov_r]
agent_attributable = moved and any(
    L1(a_prev, p) >  safety_r and L1(a_new, p) <= safety_r
    for p in observed_positions
)
```

The `observed_positions` set is exactly the set of humans whose
buffer cells `AgentController` already treats as forbidden.  The
controller refuses to step into any observed buffer; if forced, it
Safe-Waits (`moved=False`).  So the `moved and any(...)` conjunction
is vacuously false on every shipped controller, and
`agent_attr_max = 0` appears in every row of every scaling CSV not
because Theorem 1 is being verified, but because the metric is
self-fulfilling: the planner enforces its own constraint, and the
metric checks whether the planner enforced its own constraint.

**Replacement rule.** The classifier is rebuilt as a per-pair WAIT
counterfactual that does NOT consult the FOV / observed set:

> For each violation pair $(a_i, h)$ at $t+1$ (any human $h$ with
> $\ell_1(s_i(t+1), h_{\mathrm{pos\;at\;}t+1}) \le r_{\mathit{safe}}$):
> the pair is **agent-attributable** iff the agent moved AND
> $\ell_1(s_i(t), h_{\mathrm{pos\;at\;}t+1}) > r_{\mathit{safe}}$
> (WAIT would have left the agent safe vs $h$); otherwise it is
> **exogenous-attributable** (either the agent didn't move so WAIT
> was the chosen action, or WAIT would have been unsafe vs $h$
> anyway).

The rule iterates over every human within $r_{\mathit{safe}}$ of
$a_{\mathrm{new}}$ -- not the FOV-restricted set -- and assigns
attribution per pair, not per agent.  Two consequences:

1. The metric is now independent of the planner's internal state.
   A planner that knew nothing about its forbidden set and walked
   blindly into a buffer would correctly read as agent-attributable.
2. The metric can be nonzero on FOV-blind moves -- the case the old
   rule structurally missed.  Empirically the §5.4 sweeps now
   report a real nonzero `agent_attr` distribution rather than the
   construction-zero of the old rule.

**Paper-claim implication.** The pre-rewrite metric was, by
construction, the empirical witness of Theorem 1's
agent-attribution invariant: the proof says "no agent action enters
an observed buffer" and the metric checked exactly that.  Under the
new rule, `violations_agent_attributable` is no longer the empirical
witness of Theorem 1 -- it is a different (stricter) quantity.  The
Theorem 1 invariant still holds for the planner, but it is now
verified through code-walk and resolver-fallback tests
(`tests/test_theorem1_resolver.py::test_loser_fallback_respects_F`
and the controller branches in
`tests/test_theorem1_stress.py`).  The paper's Definition 1 should
be re-stated to distinguish "the planner's guarantee" (Theorem 1)
from "the measured WAIT-counterfactual" (this metric) -- they were
conflated in the original draft.

**Invariant.**  The split still satisfies
`safety_violations == violations_agent_attributable + violations_exogenous_attributable`
on every tick and at finalize; the invariant is now asserted in
`MetricsTracker.finalize` and breaks the run loud if a future edit
miscounts.

**Acceptance scenarios** (`tests/test_safety_classification.py`):

* `test_wait_counterfactual_fov_blind_move_is_agent_attributable` --
  an agent moves into the buffer of a human outside its FOV; the
  old rule scored this as exogenous (witness unobserved), the new
  rule scores it as agent-attributable (WAIT was safe).  This is
  the test that proves the new rule isn't a tautology.
* `test_wait_counterfactual_unavoidable_overlap_is_exogenous` --
  a human ends up on (or adjacent to) the agent's cell after the
  human-motion phase; WAIT was also unsafe, so the pair is
  exogenous and `agent_attributable == 0`.
* `test_attribution_invariant_holds_across_mixed_pairs` --
  one agent generates one agent-attributable pair and one
  exogenous-attributable pair in the same tick; verifies the
  per-pair split and the finalize invariant.

**Why "don't keep the old check" was the right call.** The prompt
that authorised this change ruled out a side-by-side dual metric.
Keeping the FOV-gated counter under a new name would invite
downstream readers to keep reading it as Theorem 1's empirical
witness, defeating the rewrite's whole point.  Theorem 1's
invariant is now verified by code-walk + resolver-fallback tests,
not by a metric.

## 14. Metric definitions and normalization (P6 audit)

The P6 audit flagged five issues in `core/metrics.py` that broke
cross-run comparability or counted the wrong quantity outright.  All
five are fixed; no existing CSV columns were removed (downstream plot
scripts read by name) -- the legacy names are preserved as aliases or
flagged deprecated in their docstrings, and new columns expose the
audit-corrected quantities.

### 14.1 `safety_violation_rate` mis-normalization

The legacy field divides by `total_steps` only:

```python
sv_rate = safety_violations / total_steps * 1000.0
```

But `wait_fraction` divides by `num_agents * total_steps`.  When
`num_agents` varies (every scaling sweep), the legacy rate inflates
with fleet size and is not comparable across cells.

**Fix.** New column `safety_violation_rate_per_agent_step` computes
`safety_violations / (num_agents * total_steps)`, matching the
normalization of `wait_fraction`.  The legacy `safety_violation_rate`
is kept (back-compat) and flagged **deprecated** in its docstring;
new scaling tables/plots should switch to the agent-normalized
field.  Acceptance test:
`tests/test_metrics_invariants.py::
test_safety_violation_rate_per_agent_step_is_agent_count_invariant`.

### 14.2 Loitering-human over-count -> debounced events

The per-tick classifier increments `safety_violations` and the
attribution counters once per (agent, human, tick).  A human that
loiters inside an agent's safety buffer for 10 ticks therefore
contributes 10 to `safety_violations` (and to one of the attribution
buckets) -- the metric counts agent-ticks of overlap, not events of
overlap.  This was unflagged for §5.4 tables that read the numbers
as "number of buffer breaches".

**Fix.** Three new event-debounced counters:

* `safety_violation_events` -- count of distinct overlap runs
  across all `(agent_id, human_id)` pairs.
* `violations_agent_attributable_events`
* `violations_exogenous_attributable_events`

A new MetricsTracker state machine
(`record_violation_pair` + `close_violation_tick`) tracks which
pairs were in violation in the previous tick, and bumps the event
counter only on leading edges (transition from not-in-violation to
in-violation).  The classifier in
`simulator.py::_detect_collisions_and_near_misses` calls
`record_violation_pair(aid, hid, bucket)` for each detected pair
and `close_violation_tick()` once per tick (even when the
detection block is skipped, so dropped pairs are forgotten).

Three new `*_agent_ticks` columns are added as honestly-named
aliases for the legacy per-tick counters
(`safety_violation_agent_ticks ==  safety_violations`,
`violations_agent_attributable_agent_ticks ==
violations_agent_attributable`,
`violations_exogenous_attributable_agent_ticks ==
violations_exogenous_attributable`).  Paper tables / plots pick
one of the two normalizations explicitly:

| Use                                    | Column                                       |
| -------------------------------------- | -------------------------------------------- |
| "How much overlap time across the run" | `*_agent_ticks`  (legacy field also OK)      |
| "How many buffer breaches"             | `*_events`                                   |
| Cross-fleet comparison                 | `safety_violation_rate_per_agent_step`       |

Acceptance tests:
* `test_loitering_human_yields_events_below_agent_ticks` --
  ten ticks of overlap on one pair -> agent_ticks == 10, events == 1.
* `test_distinct_events_per_pair_and_per_run` -- re-entry of the
  same pair after a quiet tick counts as a new event; two pairs
  active in the same tick count separately.

### 14.3 Lifelong-mode `makespan` is meaningless

`makespan` is the last task-completion step.  In lifelong mode a
fresh task almost always completes near `total_steps`, so
`makespan ≈ total_steps` regardless of solver / fleet / map.  The
column is preserved for one-shot mode and back-compat, but the
docstring on `Metrics.makespan` and the CSV header note are flagged
**deprecated for lifelong reporting**.  A new column
`mean_task_completion_span` (numerically identical to
`mean_flowtime`; same field, paper-aligned name) is added so
downstream plots can switch off `makespan` without a column rename
mid-paper.  Acceptance test:
`test_mean_task_completion_span_mirrors_mean_flowtime`.

### 14.4 `on_task_assigned` wrote to a non-existent attribute

```python
record.assigned_step = step
record.assigned_agent = agent_id   # <-- not a _TaskRecord field
```

Python lets a dataclass instance accept new attributes silently,
so `record.assigned_agent` quietly became an instance attribute
while the documented `record.agent_id` field stayed `None` until
`on_task_completed` overwrote it.  Callers reading `record.agent_id`
between assignment and completion saw `None` and concluded the
task was unassigned.

**Fix.** `on_task_assigned` now writes `record.agent_id = agent_id`.
Acceptance test: `test_on_task_assigned_records_agent_id`.

### 14.5 `throughput_timeline` off-by-one

The timeline bookkeeping bucketed each completion at index
`rec.completed_step` only when `0 <= rec.completed_step < total_steps`.
A task completing exactly at `total_steps` (boundary case, possible
when test code or external callers feed `on_task_completed`
directly with the final step value) was silently dropped from the
timeline while still counted in the scalar `_completed_tasks`,
producing `last_cumulative < completed_tasks`.

**Fix.** Clamp the index to `min(completed_step, total_steps - 1)`
so the timeline and the scalar count agree on the same set of
tasks.  Acceptance test:
`test_throughput_timeline_counts_match_completed_tasks_at_boundary`.
