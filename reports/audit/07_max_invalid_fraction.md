> **STATUS: SUPERSEDED by audits 09 and 11.** This report closed
> BUG #1 on the basis of a status-only retroactive check
> (`status == 'ok'` as the validity predicate). Audit 09
> reopened the bug under the YAML-verbatim predicate; audit 11
> hardened that finding on the solver clause alone (14/14
> sweeps fail, minimum invalid fraction 0.591, arithmetic hand-
> verified to 1e-12). **The "no committed result is invalidated"
> verdict in §3 of this report is incorrect.** Read
> `09_strong_validity_predicate.md` and `11_solver_fail_hardened.md`
> before acting on anything written here. The placement fix in §1
> and the enforcement wiring in §2 remain valid; only the
> retroactive verdict in §3 is superseded.

Audit step 05 BUG #1 documented that `max_invalid_fraction` was a
two-level no-op:

1. **Placement**: the runner reads `spec["max_invalid_fraction"]` at the
   top level, but every committed tuning YAML nested the field under
   `base:` at indent 2.  The top-level lookup missed it; the inner-cell
   pass warned and dropped it.
2. **Enforcement**: even if the value had been read, no downstream
   gate consumed it.  `validate_paper_claims.py` does not reference
   the field; the runner only logged the value.

This step closes both ends and runs the retroactive check the original
audit asked for.

---

## 1. Canonical location: spec top-level

The runner already reads `spec["max_invalid_fraction"]` at the top
level (`run_paper_experiment.py:689-697`).  Rather than chase the
value through every potential nesting location, this step makes the
runner's existing top-level read the **single canonical placement**
and makes any other location a load-time error.

### Wrong-placement detection

`expand_manifest` (`scripts/evaluation/run_paper_experiment.py:194-223`)
now inspects `spec["base"]` and every `groups[*].sweep` for
`max_invalid_fraction`.  If the key appears in either, it raises
`ValueError` with the canonical-placement note and a pointer to this
report.  Two messages, one per misplacement:

- Nested under `base:`:
  > spec key `max_invalid_fraction` is misplaced under `base:` — it
  > is a sweep-level threshold and must appear at the SPEC TOP LEVEL.
  > Dedent the line so it is a peer of `base:` / `seeds:` / `groups:`...
- Nested inside `groups[*].sweep`:
  > spec key `max_invalid_fraction` is misplaced inside a
  > `groups[*].sweep` cell — it is a sweep-level threshold, not a
  > per-run swept value.  Move it to the SPEC TOP LEVEL...

### YAML migration

13 committed YAMLs had the field nested under `base:` at indent 2.  A
one-off helper (`scripts/diagnostics/dedent_max_invalid.py`) walked
the contiguous comment block preceding each field declaration and
moved both together to the spec top level, between the closing of
`base:` and the `seeds:` declaration.

### Generator-side migration (the regeneration test)

The 13 tuning YAMLs are auto-generated by per-sweep generators under
`scripts/tuning/generate_*_yaml.py`.  Each generator calls the
shared helper
`scripts/tuning/_sweep_config_common.py::base_validity_guard_yaml`,
which used to emit the field with a 2-space indent (i.e. inside
`base:`).

This step:
- Changed `base_validity_guard_yaml`'s default `indent` from `"  "`
  (inside `base:`) to `""` (spec top-level).  The updated docstring
  spells out that callers must splice the result between the closing
  of `base:` and the `seeds:` declaration.
- Updated all 13 generators to splice the helper call at the new
  location (`scripts/diagnostics/migrate_validity_helper.py` +
  `scripts/diagnostics/fix_validity_splice.py` for the two-step
  rewrite + syntax fix).

`tests/test_sweep_config_dryrun.py` (39 tests) regenerates each YAML
and diffs against the committed copy; all 39 pass after the
migration, proving the generators and the on-disk YAMLs are in
lockstep.

## 2. Threshold enforcement

`run_paper_experiment.main` now consumes `spec_max_invalid_fraction`
to gate the post-sweep exit code:

```python
sweep_threshold_breached = False
if spec_max_invalid_fraction is not None and total_runs > 0:
    sweep_invalid_fraction = total_invalid / total_runs
    if sweep_invalid_fraction > spec_max_invalid_fraction:
        sweep_threshold_breached = True
        logger.error("sweep-level validity gate FAILED: ...")
    else:
        logger.info("sweep-level validity gate PASSED: ...")
```

The gate fires at the same place the existing per-(solver, map) cell
gate fires (after `write_run_validity_summary`).  Both can fire
independently; either failure routes to exit code 3 (reserved for
"sweep completed but failed validity contract", distinct from
preflight failure exit code 2 and generic exception exit code 1):

```python
if failing_cells or sweep_threshold_breached:
    return 3
return 0
```

The same conjunction is applied in the auto-stats short-circuit
branch (`run_paper_experiment.py:889`).

The sweep-level invalid predicate is the existing per-row
`_row_is_valid` (`run_paper_experiment.py:460-472`): true iff
`run_valid == True` (or the column is missing, treated as valid for
legacy compatibility).

sweep CSV under `logs/tuning/`, computes the invalid-row fraction

against the intended threshold from the corresponding YAML.

0.0 (strict) in every YAML; observed invalid fraction is 0.0 in

### Per-row corroboration

To distinguish "no `run_valid` column → vacuously valid" from "every
row truly is valid", a separate pass counted `status` values across
all sweeps:

```
$ python -c "
from collections import Counter
import csv
from pathlib import Path
for sd in sorted(Path('logs/tuning').iterdir()):
    rcsv = sd / 'results.csv'
    if not rcsv.is_file(): continue
    with rcsv.open() as f:
        rows = list(csv.DictReader(f))
    print(sd.name, dict(Counter(r.get('status') for r in rows)))
"
```

Output: every sweep reports `{'ok': N}` for its row count `N`.  No
row carries `status='error'` or any other non-ok value.

### Statement

> The disabled guard was inert but the data clears the bar anyway.
> No committed result is invalidated by the retroactive check; only
> the guard itself was untrustworthy.  Paper sections built from
> these sweeps (§5.1 horizon tuning, §5.3 FoV / safety sweep, §5.4
> scaling, §5.5 allocator comparison, §5.6 soft-safety ablation,
> §5.7 deadlock-wait) do not need to be re-run on the strength of
> this guard; they were already accidentally compliant.

This does not vacate the placement and enforcement fixes — both
fences are now load-bearing for future sweeps where the data might
not be as well-behaved.
