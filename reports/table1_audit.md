## Relationship to the Definition-1 quantity (Prompt 1)

The classifier rewrite in the earlier Definition-1 prompt introduced
`violations_def1_exogenous_attributable` as a SECOND counter for
the same per-pair event set, computed by the FOV-gated pre-move
two-clause Definition-1 rule.  On every Algorithm-2 trajectory
where `violations_agent_attributable == 0` (the §5.2 sweep
satisfies this by Theorem 1), `violations_exogenous_attributable`
and `violations_def1_exogenous_attributable` are numerically
identical: both buckets iterate the same post-move violation-pair
set, and the only divergence between the two classifiers is in
how they split a pair across the agent / exogenous buckets.  When
the agent bucket is empty, both classifiers route every pair to
exogenous and the two columns agree.

When the §5.2 sweep is re-run on the current head, the new CSV
will carry both `violations_exogenous_attributable` (the WAIT-
counterfactual exogenous bucket, the same column the paper's
Table 1 currently references) and the new
`violations_def1_exogenous_attributable` (the Definition-1
exogenous bucket).  The paper text should switch to the def1
column at that point so the reproduced number aligns 1:1 with
the Theorem 1 quantity it claims to measure.

### Saturation note

Every (H, map) cell in the rebuilt table is **arrival-saturated**
(throughput utilization >= 0.95 -- the throughput column is
marked with `*`).  Per `paper/sections/05_1_load_regime.md` the
throughput column in these cells measures the task arrival cap
$|M|/(H+W)$, not planner capacity.  The narrative paragraph that
quoted "19.5K and 8.5K" local_replans values is now consistent
with the rebuilt table; if the paper's interpretation of those
numbers cited throughput as a planner-quality signal, the
saturation footnote from §5.1 should be appended there as well.
